Visão do Produto

O que é o produto:

Trata-se de um aplicativo de pedidos de comida, voltado para conectar usuários a restaurantes locais, facilitando o processo de escolha, compra e entrega de refeições.

O que ele faz:

  • Exibe cardápio dos restaurantes.
  • Permite personalização de pedidos (ex: retirar ingrediente).
  • Suporte a múltiplas formas de pagamento.
  • Geração de histórico de pedidos.
  • Rastreamento da entrega em tempo real.

O que ele não faz:

  • Não realiza a produção dos alimentos.
  • Não gerencia estoque dos restaurantes.
  • Não atua como meio de pagamento (intermedia o processo via APIs externas).
  • Não oferece suporte presencial ou telefone.

Apresentação textual e diagramas

O projeto visa atender uma demanda crescente por conveniência e agilidade em pedidos de comida. A proposta é simples, porém escalável, com foco na experiência do usuário e na integração fluida entre clientes, restaurantes e entregadores.

Decisões Arquiteturais

  • Modularização: Separação por domínios (Usuário, Pedido, Restaurante).
  • REST API: Escolha por ser mais simples e rápida para o MVP.

  • Banco Relacional: PostgreSQL, pela robustez em modelagem de relacionamentos.

  • Cloud First: Adoção de serviços SaaS e PaaS para reduzir esforço de infraestrutura.
  • Auth externa: Redução de esforço com autenticação utilizando Firebase ou Auth0.

Modelos de Arquitetura

Arquitetura em camadas com base no Clean Architecture adaptado:

1. Separação de responsabilidades

Cada camada tem uma responsabilidade clara:

  • Frontend: Interação com o usuário.
  • Camada de Aplicação (API Gateway): Coordenação das ações e controle de fluxo.
  • Camada de Domínio (Serviços e Regras de Negócio): Lógica pura e regras do negócio.
  • Infraestrutura: Comunicação com o banco de dados, APIs externas, mensageria, etc.

Isso permite que mudanças em uma camada causem impacto mínimo nas outras.

2. Facilidade de testes

A lógica de negócio fica isolada do framework e da infraestrutura, permitindo testes unitários simples e rápidos.

Exemplo: podemos testar a lógica de um pedido sem precisar de banco de dados ou API externa.

3. Escalabilidade

A arquitetura modular por domínios permite que módulos cresçam de forma independente:

  • Podemos escalar o módulo de pedidos separadamente do módulo de usuários.
  • Facilita a transição futura para microsserviços, se necessário.

4. Flexibilidade tecnológica

A arquitetura proposta não amarra o projeto a uma tecnologia específica:

  • O frontend pode ser React ou Flutter Web.
  • O backend pode migrar de REST para GraphQL.
  • A camada de infraestrutura pode trocar PostgreSQL por MongoDB sem alterar regras de negócio.

 5. Aderência ao DDD (Domain-Driven Design)

A organização em torno de "domínios" como Pedidos, Restaurantes, Usuários e Entregas segue os princípios do DDD, facilitando a modelagem alinhada ao problema real.

6. Preparada para mensageria

A separação clara entre módulos e a previsibilidade das interações tornam fácil a inclusão de RabbitMQ/Kafka, importante para:

  • Notificações em tempo real.
  • Processos assíncronos (como rastreamento de entrega).
  • Integrações desacopladas.

7. Próxima da Clean Architecture, mas adaptada para agilidade

A Clean Architecture pura pode ser excessivamente complexa para MVPs. Esta versão é simplificada para entregar rápido, mas ainda mantém os pilares essenciais da escalabilidade e testabilidade.
